## 1. 먼저 알아야 하는 생각법

Verilog는 소프트웨어처럼 "위에서 아래로 한 줄씩 실행되는 언어"로 보면 계속 헷갈린다.  
실제로는 하드웨어 구조를 기술하는 언어에 가깝다.

핵심은 아래 두 가지를 먼저 구분하는 것이다.

1. 조합논리
현재 입력이 바뀌면 출력이 바로 바뀌는 회로다. 저장 기능이 없다.

2. 순차논리
클럭 edge에서만 값이 갱신되는 회로다. 저장 기능이 있다.

실무에서는 대부분의 버그가 아래에서 나온다.

- 조합논리인데 출력 할당을 빠뜨려 latch가 생김
- 순차논리인데 `=` 와 `<=` 를 섞어 써서 동작을 잘못 이해함
- 비동기 입력을 동기화하지 않고 바로 사용함
- FSM을 한 블록에 다 몰아넣어서 유지보수가 어려워짐

## 3. `reg` 와 `wire` 를 먼저 정리

처음 배우면 `reg = 저장소`, `wire = 선`처럼 외우기 쉽다.  
그런데 그렇게만 이해하면 나중에 틀리기 쉽다.

좀 더 정확하게는 이렇게 보는 편이 낫다.

- `wire`
연결되는 신호다. 보통 `assign` 이나 모듈 출력 연결에 많이 쓰인다.

- `reg`
`always` 또는 `initial` 블록 안에서 값을 할당받는 변수형이다.

중요한 점:

- `reg` 라고 해서 무조건 플립플롭이 되는 것은 아니다.
- 조합논리 `always @(*)` 안에서 할당하면 조합논리 출력용 `reg` 가 될 수 있다.
- 순차논리 `always @(posedge clk)` 안에서 할당하면 플립플롭 성격을 갖게 된다.

즉, 타입 이름보다 "어떤 always 블록에서 어떻게 할당되느냐"가 더 중요하다.


## 4. 조합논리의 메커니즘

대상 모듈: `simple_alu`

조합논리는 현재 입력만으로 현재 출력을 결정한다.  
기억 기능이 없어서 "이전 값"을 보관하지 않는다.

### 동작 메커니즘

예를 들어 ALU에서 `a=10`, `b=3`, `op=add` 라고 하자.

1. 입력 `a`, `b`, `op` 중 하나라도 바뀌면 `always @(*)` 블록이 다시 계산된다.
2. 블록 시작에서 `y=0`, `carry=0` 기본값을 먼저 넣는다.
3. `case(op)` 에 따라 add, sub, and, or 같은 연산 중 하나를 선택한다.
4. 선택된 결과가 바로 출력으로 반영된다.

이 회로를 알고리즘처럼 표현하면 아래와 같다.

```text
입력이 바뀌면:
  y = 0
  carry = 0

  op 값을 보고:
    add면 a+b
    sub면 a-b
    and면 a&b
    or면 a|b
    xor면 a^b
    shift면 비트 이동
```

### 왜 기본값이 중요한가

기본값이 없고 일부 분기에서만 `y` 를 할당하면, 합성 도구는 "할당되지 않은 경우 이전 값을 유지해야 한다"고 해석할 수 있다.  
그러면 원치 않는 latch가 생긴다.

즉 조합논리의 핵심 규칙은 이것이다.

- 모든 출력은 항상 결정되어야 한다.


## 5. 순차논리의 메커니즘

대상 모듈: `counter_with_enable`

순차논리는 "시간"이 들어간다.  
보통 클럭 상승엣지에서만 상태가 바뀐다.

### 동작 메커니즘

이 카운터는 매 클럭마다 아래 순서로 생각하면 된다.

```text
클럭 상승엣지가 오면:
  reset이 0이면 count를 0으로 만든다.
  reset이 아니고 en이 1이면 count를 1 증가시킨다.
  아니면 count를 유지한다.
```

소프트웨어처럼 while 문이 도는 것이 아니다.  
하드웨어 플립플롭이 매 클럭마다 값을 샘플링한다고 생각해야 한다.

### 왜 `<=` 를 쓰는가

순차논리에서는 여러 레지스터가 같은 클럭 edge에서 동시에 업데이트되는 모델이 필요하다.  
그래서 `non-blocking assignment(<=)` 를 쓴다.

실무 규칙은 거의 고정이라고 보면 된다.

- 조합논리: `=`
- 순차논리: `<=`


## 6. 비동기 입력 동기화의 메커니즘

대상 모듈: `sync_and_rise_detect`

이 부분은 실무에서 매우 중요하다.  
버튼 입력, 센서 입력, 다른 클럭 도메인 신호는 현재 클럭 기준으로 보면 "언제 바뀔지 모르는 신호"다.

이런 신호를 바로 플립플롭이나 FSM 조건으로 쓰면 메타안정성 문제가 생길 수 있다.

### 2단 동기화 알고리즘

```text
매 클럭 상승엣지마다:
  sync_ff1 <= async_in
  sync_ff2 <= sync_ff1
  sync_ff2_d <= sync_ff2
```

이 구조의 의미:

1. 첫 번째 플립플롭 `sync_ff1`
비동기 입력을 처음 받아들인다.

2. 두 번째 플립플롭 `sync_ff2`
한 번 더 안정화된 값을 내부 로직에서 쓰게 만든다.

3. 이전값 저장 `sync_ff2_d`
이전 클럭의 값을 기억해서 edge 검출에 사용한다.

### 상승엣지 검출 메커니즘

```text
pulse = 현재값이 1이고, 이전값이 0일 때만 1
```

즉,

- `sync_ff2 = 1`
- `sync_ff2_d = 0`

이면 `pulse = 1` 이 된다.

이 신호는 보통 1클럭 폭의 짧은 펄스가 된다.


## 7. FSM의 메커니즘

대상 모듈: `simple_fsm`

FSM은 상태를 가지고 행동하는 회로다.  
소프트웨어의 `switch-case` 와 비슷해 보이지만, 실제로는 상태 저장용 레지스터와 조합논리가 결합된 구조다.

이 예제의 상태는 세 가지다.

- `S_IDLE`
- `S_RUN`
- `S_DONE`

### 실무에서 FSM을 3부분으로 나누는 이유

1. 현재 상태 저장 블록
클럭 edge에서 `state <= next_state`

2. 다음 상태 계산 블록
현재 상태와 입력을 보고 `next_state` 결정

3. 출력 계산 블록
현재 상태에 따라 `busy`, `done` 출력 결정

### 상태 전이 알고리즘

```text
현재 상태가 IDLE이면:
  start가 1일 때 RUN으로 이동

현재 상태가 RUN이면:
  work_done이 1일 때 DONE으로 이동

현재 상태가 DONE이면:
  1클럭 뒤 IDLE로 복귀
```

### 왜 이렇게 나누는가

이렇게 나누면 아래가 쉬워진다.

- 상태 전이 디버깅
- 출력이 왜 올라갔는지 분석
- 상태 추가 및 수정
- 타이밍 이슈 추적

FSM을 처음부터 한 always 블록에 몰아 넣는 습관은 피하는 것이 좋다.


## 8. latch가 생기는 메커니즘

대상 모듈: `bad_example_latch`

문제 코드는 아래 개념이다.

```text
if (sel) y = a + b;
```

그런데 `sel=0` 일 때 `y` 값이 정의되지 않는다.

합성 도구 입장에서는 이렇게 보인다.

```text
sel이 0일 때는 이전 y 값을 유지해야 하네?
그럼 저장소가 필요하네?
```

그래서 latch를 만들 수 있다.

### latch를 피하는 규칙

조합논리에서는 항상 아래 둘 중 하나를 지켜야 한다.

1. 블록 시작에서 기본값을 할당한다.
2. 모든 분기에서 빠짐없이 값을 할당한다.

`good_example_no_latch` 는 그 규칙을 지킨 버전이다.


## 9. `=` 와 `<=` 의 실제 차이

대상 모듈: `blocking_vs_nonblocking`

이 예제는 왜 순차논리에서 `<=` 를 써야 하는지 감각을 주는 예제다.

### 현재 코드의 의미

```verilog
q1 <= d;
q2 <= q1;
```

같은 클럭 edge에서 실행되므로,

- `q1` 은 현재 `d` 를 받는다.
- `q2` 는 업데이트되기 전의 `q1` 을 받는다.

즉 결과적으로 2단 레지스터 체인처럼 동작한다.

### 왜 중요한가

만약 순차논리에서 `=` 를 섞어 쓰면, 시뮬레이터에서 보이는 계산 순서를 사람이 잘못 이해하기 쉽다.  
실무에서는 이 부분 때문에 디버깅 시간을 많이 낭비한다.


## 10. 테스트벤치의 메커니즘

대상 모듈: `tb_verilog_practical_starter`

테스트벤치는 하드웨어를 "만드는 코드"가 아니라 "검증하는 코드"다.

### 테스트벤치의 기본 알고리즘

```text
1. 클럭 생성
2. reset 인가
3. 입력 자극(stimulus) 제공
4. 출력 관찰
5. 기대 동작과 비교
6. 종료
```

이 예제에서는 다음을 한다.

- `forever #5 clk = ~clk;` 로 클럭 생성
- 초기에 `rst_n=0` 으로 reset 유지
- ALU 입력을 바꿔가며 연산 결과 확인
- counter의 `en` 을 켜서 증가 확인
- async 입력을 클럭과 어긋나게 바꿔 동기화 확인
- FSM의 `start`, `work_done` 을 넣어 상태 전이 확인

### 짧은 pulse 신호를 어떻게 봐야 하는가

`pulse`, `done` 같은 신호는 1클럭만 올라갈 수 있다.  
그래서 나중에 한 번 `display` 하면 이미 0으로 내려가 있을 수 있다.

그래서 아래처럼 이벤트 순간을 잡아서 확인하는 방식이 좋다.

- `always @(posedge pulse)`
- `always @(posedge clk) if (done) ...`


## 11. 파형 보는 법

이 부분은 처음 배울 때 매우 중요하다.  
텍스트 출력보다 파형을 보면 회로의 시간 흐름이 훨씬 잘 보인다.

### 준비물

보통 아래 도구를 쓴다.

- `iverilog`
- `vvp`
- `gtkwave`

역할은 이렇다.

- `iverilog`
Verilog 코드를 컴파일한다.

- `vvp`
컴파일된 시뮬레이션 실행 파일을 돌린다.

- `gtkwave`
생성된 `.vcd` 파형 파일을 시각적으로 보여준다.

### 현재 예제에 이미 넣어둔 파형 덤프 코드

테스트벤치 안에 아래 코드가 들어 있다.

```verilog
initial begin
    $dumpfile("verilog_practical_starter.vcd");
    $dumpvars(0, tb_verilog_practical_starter);
end
```

의미:

- `verilog_practical_starter.vcd` 라는 파형 파일을 만든다.
- 테스트벤치 아래 신호 변화를 기록한다.

### 실행 절차

같은 디렉터리에서 아래 순서로 실행하면 된다.

```bash
iverilog -o verilog_practical_starter.out verilog_practical_starter.v
vvp verilog_practical_starter.out
gtkwave verilog_practical_starter.vcd
```

### 파형에서 먼저 봐야 할 신호

처음에는 신호를 너무 많이 보지 말고 아래만 먼저 보자.

- `clk`
- `rst_n`
- `a`
- `b`
- `op`
- `alu_y`
- `en`
- `count`
- `async_in`
- `pulse`
- `start`
- `work_done`
- `busy`
- `done`

### 파형 읽는 방법

1. `clk` 와 `rst_n` 을 먼저 본다.
회로가 언제 리셋 상태인지 알아야 다른 신호를 해석할 수 있다.

2. ALU 구간을 본다.
`op` 값이 바뀔 때 `alu_y` 가 바로 바뀌는지 확인한다.

3. counter 구간을 본다.
`en=1` 인 동안만 `count` 가 클럭마다 증가하는지 확인한다.

4. sync 구간을 본다.
`async_in` 이 애매한 시점에 바뀌더라도 `pulse` 는 클럭 기준으로 정리된 짧은 펄스로 나오는지 본다.

5. FSM 구간을 본다.
`start` 이후 `busy` 가 올라가고, `work_done` 이후 `done` 이 1클럭 올라오는지 확인한다.

### 파형을 볼 때 초보자가 자주 하는 실수

- 조합논리와 순차논리를 같은 방식으로 해석함
- edge 기준 신호를 레벨 기준으로 봄
- reset 구간을 무시하고 결과만 봄
- 짧은 pulse를 놓치고 "동작 안 한다"고 판단함


## 12. 실무에서 꼭 기억할 규칙

아래는 실제로 계속 쓰게 되는 체크리스트다.

### 조합논리 체크리스트

- `always @(*)` 를 쓰고 있는가
- 모든 출력에 기본값이 있는가
- 모든 분기가 출력을 정의하는가
- latch가 생길 여지가 없는가

### 순차논리 체크리스트

- `always @(posedge clk ... )` 구조가 명확한가
- 레지스터 할당에 `<=` 를 쓰고 있는가
- reset 동작이 분명한가
- enable 조건이 의도대로 동작하는가

### 인터페이스 체크리스트

- 외부 입력은 동기화했는가
- 다른 클럭 도메인 신호를 바로 쓰고 있지 않은가
- 1클럭 pulse 신호를 놓치지 않게 설계했는가

### 검증 체크리스트

- 최소한의 테스트벤치가 있는가
- 파형을 보고 시간축 기준으로 검증했는가
- 기대값과 실제값을 비교했는가

## 14. 한 줄 요약

Verilog 입문에서 가장 중요한 것은 "문법"보다 "이 코드가 조합논리인지, 순차논리인지, 상태를 기억하는지, 클럭 기준으로 언제 바뀌는지"를 읽어내는 능력이다.
